# Polish Translation (i18n) — Design

**Date:** 2026-04-13
**Status:** Approved (design phase)
**Scope:** Add a Polish (`/pl/`) localized version of the site, covering UI chrome, static pages, companies, and projects. Blog posts, publications, glossary, photography, and changelog stay English-only. Translations are produced by Claude in local Claude Code sessions on demand, gated by a drift-detection script.

## Goals

- A Polish-speaking visitor lands on `/pl/` and gets a fully localized experience for the marketing surface (homepage, about, CV, work, contact, company pages, project pages).
- The English site keeps its current URLs and SEO authority — Polish is purely additive under `/pl/`.
- Polish URLs use translated slugs (`/pl/o-mnie`, `/pl/praca/gyfted`) per Google's international SEO guidance.
- New or changed English content is detected automatically; Polish translations are produced on demand by Claude (not by an unattended CI job).
- Voice and quality on long-form prose is preserved by keeping a human review gate (Blazej reviews the diff before commit).
- Blog posts, publications, glossary, photography, and changelog are explicitly excluded.

## Non-Goals

- Translating blog posts, publications, glossary entries, photography pages, or the changelog.
- Auto-detecting browser language and redirecting users.
- Localizing currency, date, or number formats (the site has none).
- Right-to-left support.
- Building a CMS, translation memory database, or per-paragraph stable-ID system.
- Running translations unattended in CI (no GitHub Action, no API key in CI).
- Treating drift as a blocking error in CI.

## Decisions Made During Brainstorming

| # | Question | Decision |
|---|---|---|
| 1 | Scope of translation | UI chrome + static pages (`about`, `cv`, `work`, `contact`, homepage, 404) + companies (4) + projects (6). Glossary, blog, publications, photography, changelog excluded. |
| 2 | Translation mechanism | Auto for UI strings; auto-draft + Blazej's review for long-form prose. |
| 3 | URL structure | Subpath, English at root: `/about` (EN), `/pl/o-mnie` (PL). |
| 4 | Language switcher | Header switcher, smart fallback to `/pl/` (or `/`) when no per-page equivalent exists. |
| 5 | Automation trigger | Originally GitHub Action → **revised to:** Claude in local Claude Code sessions, invoked ad hoc in conversation. |
| 6 | Source-of-truth / drift handling | Hash-tracked. Each `.pl.md` carries `source_hash`. Drift detected by script; reconciliation by Claude with Blazej review. |
| 7 | Dev/rollout strategy | Git worktree (`feat/polish-i18n` branch), draft PR to `main`, Netlify deploy preview as the isolated test environment, merge when satisfied. |
| 8 | Slug translation | Translated slugs: `/pl/o-mnie`, `/pl/praca`, `/pl/kontakt`. `cv` stays as `cv`. Project/company slugs (proper nouns) stay unchanged. |

## Architecture Overview

Three moving parts:

1. **i18n dictionary** — `src/i18n/en.json` and `src/i18n/pl.json` hold all UI chrome strings (nav, footer, buttons, form labels, 404, SEO defaults). Components read via a `t(key, locale)` helper. `en.json` is hand-maintained; `pl.json` is generated by Claude when synced.

2. **Translatable content collections** — Polish versions live as `.pl.md` siblings: `about.md` ↔ `about.pl.md`. Each `.pl.md` carries `lang: pl` and `source_hash: <sha>` frontmatter, plus a `slug:` override when the URL slug is translated. In scope: `pages/`, `companies/`, `projects/`. Out of scope: `blog/`, `glossary/`, `photography/`, `publications.yml`, `changelog/`.

3. **Drift detection + manual sync** — `npm run translate:check` scans EN sources, compares hashes against PL siblings, prints a structured report of stale/missing translations and dictionary drift. When Blazej asks Claude to "sync polish translations" in a session, Claude runs the script, reads the drifted files, writes/updates the PL versions, bumps the hashes, and the user reviews the diff before committing.

**No CI, no API keys, no auto-PRs.** The "automation" is in detection and workflow, not unattended execution.

| `/cv` | `/pl/cv` |
| `/work` | `/pl/praca` |

| `/work/contentforge` | `/pl/praca/contentforge` |
| `/contact` | `/pl/kontakt` |
| `/404` | `/pl/404` |

Project and company slugs (proper nouns) are unchanged across locales.

### Slug translation declaration

**Route prefixes** live in `src/i18n/routes.json`:

```json
{
  "en": { "about": "about", "work": "work", "contact": "contact", "cv": "cv" },
  "pl": { "about": "o-mnie", "work": "praca", "contact": "kontakt", "cv": "cv" }
}
```

This is the single source of truth for section-level URLs. Language switcher, sitemaps, hreflang tags, and route generation all read from it.

**Per-page slugs** for content collection entries live in `.pl.md` frontmatter as a `slug:` field. When omitted, Astro uses the filename-derived slug (the EN identifier). For company/project pages with proper-noun slugs, `slug:` is omitted.

### Content schema changes

In `src/content.config.ts`, add two optional fields to the `pages`, `companies`, and `projects` collection schemas:

```ts
lang: z.enum(['en', 'pl']).default('en'),
source_hash: z.string().optional(),
```

For `pages` only, also add an optional `slug` override:

```ts
slug: z.string().optional(),
```

(Companies and projects use file-derived slugs unchanged.)

### Astro pages structure

```
src/pages/
  index.astro              → /
  about.astro              → /about
  cv.astro                 → /cv
  contact.astro            → /contact
  404.astro                → /404
  work/index.astro         → /work
  work/[slug].astro        → /work/gyfted, etc.
  pl/
    index.astro            → /pl/
    [slug].astro           → /pl/o-mnie, /pl/cv, /pl/kontakt (single segment)
    404.astro              → /pl/404
    praca/
      index.astro          → /pl/praca
      [slug].astro         → /pl/praca/gyfted, etc.
```

The single-segment `[slug].astro` under `/pl/` handles all top-level Polish pages by reading `routes.json` to map Polish slugs back to `pages/` collection entries. Astro's routing prefers more-specific files (`index.astro`, `404.astro`) over the dynamic `[slug].astro`, so they coexist cleanly. Adding a new EN page only requires updating `routes.json` and creating its `.pl.md` sibling; the dynamic route picks it up.

The `pl/praca/` folder is explicit because the route prefix itself is translated and Astro's routing is path-based.

### Slug-from-frontmatter handling

The content loader sees `about.pl.md` as an entry with id `about.pl`. Dynamic routes' `getStaticPaths()` strip `.pl` from the id and use the `slug` frontmatter field if present, otherwise the stripped id, to compute the final URL.

```json
  },

    "title": "Page not found",
    "body": "The page you're looking for doesn't exist."

```

Flat-ish, one level of namespacing. No deeper nesting — keys stay greppable.

The dictionary above is illustrative; the actual key set will be determined by the dictionary-extraction step during implementation (see below).

### The `t()` helper

`src/i18n/t.ts`:

```ts
import en from './en.json';
import pl from './pl.json';

const dictionaries = { en, pl } as const;
export type Locale = keyof typeof dictionaries;

export function t(key: string, locale: Locale = 'en'): string {
  const parts = key.split('.');
  let value: any = dictionaries[locale];
  for (const part of parts) value = value?.[part];
  if (typeof value !== 'string') {
    if (locale !== 'en') return t(key, 'en');  // fall back to EN
    return key;  // last resort: surface the key so missing strings are obvious
  }
  return value;
}
```

Three properties:
1. **EN fallback** — if a PL key is missing, returns the EN string (page renders, missing key is visible during dev).
2. **Last-resort key surfacing** — if the key is missing in both dictionaries, the literal key string renders, making typos obvious.
3. **No runtime locale detection** — locale is always passed explicitly.

### Component pattern

Components that render UI text receive a `locale` prop:

```astro
---
// src/components/Header.astro
import { t } from '../i18n/t';
import routes from '../i18n/routes.json';
interface Props { locale?: 'en' | 'pl'; }
const { locale = 'en' } = Astro.props;
const prefix = locale === 'pl' ? '/pl' : '';
---
<nav>
  <a href={`${prefix}/${routes[locale].about}`}>{t('nav.about', locale)}</a>
  <a href={`${prefix}/${routes[locale].work}`}>{t('nav.work', locale)}</a>
</nav>
```

Pages pass their locale down:

```astro
---
// src/pages/pl/index.astro
import Header from '../../components/Header.astro';
---
<Header locale="pl" />
```

### Components that don't need translation

Components that only render on EN-only surfaces (blog, publications, photography) stay untouched: `BlogSidebar`, `PostCard`, `PublicationEntry`, `PhotoGrid`, `LabelFilter`, `TagFilter`, `YearFilter`, `AlbumCard`, `PhotoItem`. Any component that might later appear on a Polish page receives the `locale` prop now.

### Dictionary-extraction step (one-time, part of implementation)

1. Grep every `.astro` file under `src/components/` and `src/pages/` for hardcoded English strings (text content, `aria-label`, `title`, `alt`, `placeholder`).
2. For each one, add a key to `en.json` and replace the literal in the component with `{t('key', locale)}`.
3. Components that don't need translation (above list) are skipped.

This is mechanical but tedious — estimated 50–100 strings across the site. Implementation plan will batch this work.

### Discipline going forward

Any new hardcoded UI string added to a component must go in `en.json` first. PL gets filled in next time translations are synced.

## Drift Detection & Translation Workflow

### The drift-detection script

`scripts/translate-check.mjs`:

1. Reads `routes.json` to know which collections are in scope.
2. Walks each in-scope collection (`pages`, `companies`, `projects`) and collects every `*.md` file that is not `*.pl.md`.
3. For each EN file, computes a SHA-256 hash (truncated to 12 chars) of its normalized content (frontmatter + body):
   - Trim trailing whitespace per line
   - Collapse final newlines
   - Sort frontmatter keys alphabetically before hashing
4. Looks for the matching `.pl.md` sibling.
5. Compares `source_hash` in PL frontmatter against the freshly computed EN hash.
6. Also checks `en.json` vs `pl.json`: every EN key should exist in PL with a matching value-shape.
7. Prints a structured report:

```
Polish translation drift report
================================

MISSING (no Polish version yet):
  src/content/projects/new-aggregator.md
  src/content/companies/swps-university.md

DRIFTED (source changed since last translation):
  src/content/pages/about.md
    PL hash: a1b2c3d4e5f6
    EN hash: 9f8e7d6c5b4a

UI DICTIONARY:
  Missing in pl.json: nav.glossary, footer.newsletter (2 keys)
  Drifted (EN value changed): post.read_more

UP TO DATE:
  src/content/pages/cv.md
  src/content/pages/work.md
  ...

Summary: 2 missing, 2 drifted, 14 up to date, 3 dictionary changes
```

The script has no dependencies — uses Node's built-in `crypto`, `fs`, `path`. Exits 0 always (informational, not gating).

### `package.json` script

```json
{
  "scripts": {
    "translate:check": "node scripts/translate-check.mjs"
  }
}
```

### The translation workflow

1. User starts a Claude Code session in the worktree (during dev) or main checkout (post-launch).
2. User says: *"sync polish translations"* (or any equivalent natural-language phrasing).
3. Claude reads `feedback_copywriting.md` and `user_writing_voice_profile.md` from memory at session start to ground voice.
4. Claude runs `npm run translate:check` and reads the report.
5. For each missing or drifted EN file, Claude:
   - Reads the EN file.
   - If a PL file exists (drift case), reads it too — to preserve hand-edits and use prior translation as voice reference.
   - Writes/updates the `.pl.md` file with the new translation and `source_hash: <new EN hash>`.
6. For UI dictionary drift, Claude updates `pl.json` to match the new/changed `en.json` keys.
7. Claude shows a summary of changes.
8. User reviews the diff in their editor (`git diff`).
9. User commits when satisfied. If changes are needed, user tells Claude what to fix.

### Drift reconciliation rules

When EN moved but PL was previously hand-edited:

- If the existing PL file's content looks auto-generated (literal translation of the previous EN), Claude overwrites freely.
- If the existing PL file shows hand-editing (notably non-literal phrasing, restructured paragraphs, voice flourishes), Claude only patches paragraphs corresponding to the EN diff and tells the user explicitly: *"about.pl.md looks hand-edited; I only updated the paragraph corresponding to the EN change in lines 14–18. Please review."*

This is judgment, not a rigid rule. The escape hatch is always "user reviews the diff and tells Claude what to do instead."

### The "skip a drift without retranslating" workflow

When EN changes are trivial and the user doesn't want to touch PL:

- User tells Claude: *"skip drift on about.md, just bump the hash"* — Claude updates `source_hash` in `about.pl.md` to the current EN hash, leaves content alone, commits. One-line edit.
- Or user does it by hand-editing the frontmatter.
- Either way, the next drift check shows "up to date" without re-translation.

### Optional: post-commit nudge

A `.git/hooks/post-commit` script (committed as `scripts/hooks/post-commit` and installed via a one-time `npm run install-hooks`) runs `translate:check` silently and, if drift is detected, prints a one-line reminder:

```
[i18n] 2 files have drifted Polish translations. Run "sync polish translations" in Claude Code to update.
```

It does no work, opens nothing, blocks nothing. Optional; can defer to a later iteration.

### What this section does NOT include

- No GitHub Action.
- No CI integration.
- No Claude API key handling.
- No auto-PR creation.
- No automated label workflows.
- No background processes.

## Language Switcher, SEO, Sitemaps & Rollout

### Language switcher component

`src/components/LanguageSwitcher.astro`:

```astro
---
import { t } from '../i18n/t';
interface Props {
  currentLocale: 'en' | 'pl';
  currentPath: string;
  alternateUrl?: string;  // explicit equivalent on translated pages; undefined on blog/publications/etc.
}
const { currentLocale, alternateUrl } = Astro.props;

function deriveAlternate(): string {
  if (alternateUrl) return alternateUrl;
  // Smart fallback: untranslated pages (blog, publications, glossary, photography, changelog)
  return currentLocale === 'en' ? '/pl/' : '/';
}
---
<div class="lang-switcher">
  {currentLocale === 'en' ? (
    <a href={deriveAlternate()} hreflang="pl" lang="pl" aria-label={t('lang_switcher.switch_to_pl', 'en')}>PL</a>
  ) : (
    <a href={deriveAlternate()} hreflang="en" lang="en" aria-label={t('lang_switcher.switch_to_en', 'pl')}>EN</a>
  )}
</div>
```

The visible label is the short code (`PL` / `EN`); the `aria-label` uses the `lang_switcher` dictionary keys for screen readers.

**Smart fallback rule:** On translated pages, the page passes an explicit `alternateUrl` derived from `routes.json` and the page's slug. On untranslated pages (blog posts, publications, glossary, photography, changelog), no `alternateUrl` is passed → falls back to `/pl/` or `/`. Switcher always works, never 404s.

### How pages compute their alternate URL

```astro
---
// src/pages/about.astro
import routes from '../i18n/routes.json';
const alternateUrl = `/pl/${routes.pl.about}`;  // → /pl/o-mnie
---
<Header locale="en" />
<LanguageSwitcher currentLocale="en" currentPath={Astro.url.pathname} alternateUrl={alternateUrl} />
```

For dynamic content collection routes, the same logic applies but reads the matching PL entry's slug from the collection.

### SEO — hreflang tags

In `src/components/SEO.astro`, add hreflang `<link>` tags to every page's `<head>`:

```astro
---
const { currentLocale, alternateUrl } = Astro.props;
const enUrl = currentLocale === 'en' ? Astro.url.href : new URL(alternateUrl, Astro.site).href;
const plUrl = currentLocale === 'pl' ? Astro.url.href : (alternateUrl ? new URL(alternateUrl, Astro.site).href : null);
---
<link rel="canonical" href={Astro.url.href} />
<link rel="alternate" hreflang="en" href={enUrl} />
{plUrl && <link rel="alternate" hreflang="pl" href={plUrl} />}
<link rel="alternate" hreflang="x-default" href={enUrl} />
```

Three rules:

1. **Self-canonical.** Each page is canonical to its own URL — never canonical to the EN version. Cross-language canonicals would tell Google the PL page is a duplicate of EN, killing PL indexing.
2. **Bidirectional hreflang.** EN page references PL with `hreflang="pl"`, PL page references EN with `hreflang="en"`. Both sides must reference each other or Google ignores the hreflang.
3. **`x-default` → English.** Tells Google "if the user's language doesn't match either, send them to EN."

### Pages without a Polish translation

Blog posts, publications, glossary, photography, changelog — emit only `hreflang="en"` and `hreflang="x-default"`, no PL alternate. Cleanest signal to Google: this content exists only in English.

### Localized title and meta description

- For content collection pages: from each page's own frontmatter (`title`, `description`), translated as part of the `.pl.md` content.
- For static `.astro` pages: from `pl.json` keys (`meta.about_title`, `meta.about_description`, etc.) added to the dictionary as needed.

### `<html lang="...">`

Set by `Base.astro` layout from a `locale` prop passed by every page.

### Open Graph and JSON-LD localization

- `og:locale` → `en_US` or `pl_PL` per page
- `og:locale:alternate` → the other locale
- JSON-LD `@language` field set to current locale
- Person/Organization JSON-LD entities: keep core fields (name, URL, sameAs) language-neutral; localize `description` and `jobTitle` per page locale

### Sitemaps

The project uses custom sitemap files (not `@astrojs/sitemap`):

- `sitemap-blog.xml.ts`, `sitemap-glossary.xml.ts`, `sitemap-photography.xml.ts`, `sitemap-changelog.xml.ts` — emit only EN URLs (no PL counterparts), no hreflang annotations. **No changes.**
- `sitemap-pages.xml.ts` and `sitemap-work.xml.ts` — gain Polish entries with hreflang annotations. **Modified in place.**
- `sitemap-index.xml.ts` — no changes (same files, more entries).

Each `<url>` entry that has a Polish counterpart gets nested `<xhtml:link rel="alternate" hreflang="..."/>` tags. Both EN and PL URLs appear as separate `<url>` entries, each cross-referencing the other:

```xml
<url>
  <loc>https://www.blazejmrozinski.com/about</loc>
  <xhtml:link rel="alternate" hreflang="en" href="https://www.blazejmrozinski.com/about"/>
  <xhtml:link rel="alternate" hreflang="pl" href="https://www.blazejmrozinski.com/pl/o-mnie"/>
  <xhtml:link rel="alternate" hreflang="x-default" href="https://www.blazejmrozinski.com/about"/>
</url>
<url>
  <loc>https://www.blazejmrozinski.com/pl/o-mnie</loc>
  <xhtml:link rel="alternate" hreflang="en" href="https://www.blazejmrozinski.com/about"/>
  <xhtml:link rel="alternate" hreflang="pl" href="https://www.blazejmrozinski.com/pl/o-mnie"/>
  <xhtml:link rel="alternate" hreflang="x-default" href="https://www.blazejmrozinski.com/about"/>
</url>
```

`robots.txt` is unchanged — already points to the sitemap index.

### Rollout plan

1. Create worktree: `git worktree add ../blazejmrozinski.com-pl feat/polish-i18n` (new branch off `main`).
2. Build the i18n machinery in the worktree: `routes.json`, dictionary files, `t()` helper, schema changes, catch-all route, language switcher, hreflang in SEO component, sitemap updates, drift script, optional post-commit hook.
3. Run the dictionary-extraction step on existing components.
4. Translate the UI dictionary (`pl.json`) — Claude, in a session in the worktree.
5. Translate the ~16 in-scope pages — Claude, in 2–3 batched sessions. Blazej reviews each batch before moving on.
6. Open a draft PR to `main`. Netlify builds a deploy preview at `deploy-preview-N--<sitename>.netlify.app`.
7. Click-test on the preview URL: every Polish page, language switcher on every page type (translated, blog post, publications, glossary, photography), hreflang in page source, sitemap fetches.
8. Iterate until satisfied. Push fixes to the branch; preview rebuilds.
9. Merge the PR. Production deploys.
10. Remove the worktree: `git worktree remove ../blazejmrozinski.com-pl`.
11. Refetch updated sitemap in Google Search Console.

### Post-launch operating mode

- User edits EN content as before. Drift accumulates silently.
- Either (a) the post-commit nudge surfaces drift, (b) the user proactively asks Claude to sync, or (c) the user ignores drift indefinitely if EN changes are minor.
- When syncing, Claude runs the drift script and does the work; user reviews the diff and commits.

### Testing strategy

- **Build-time validation** — Astro fails the build if a `.pl.md` references a non-existent slug, if `routes.json` is missing keys referenced by a component, or if a page lacks a locale. Schema enforcement does most of this.
- **Drift script as a sanity check** — informational only. Not run in CI; CI must not enforce zero drift, since that would block merges of EN-only changes.
- **Manual click-test plan** for the deploy preview — included as a checklist in the implementation plan.

## Trade-offs Accepted

- **Mechanical duplication of route prefixes** (explicit `pl/praca/` folder) instead of an Astro i18n integration. For 6 pages, the simplicity wins; an integration can be added later if scope grows.
- **Catch-all `[...slug].astro` for single-segment Polish pages** instead of one file per page. Avoids per-page file maintenance at the cost of one extra layer of indirection in routing.
- **Whole-file hash diffing** instead of per-paragraph stable IDs. Simpler, but means a single typo in EN flags the whole PL file as drifted (mitigated by Claude's reconciliation judgment and the manual hash-bump escape hatch).
- **Verbose `t('nav.about', locale)` calls** in components instead of inline strings. The verbosity is the point — translatable strings are visible in the source.
- **No CI enforcement** of translation freshness. Drift can accumulate; the trade is full user control and zero CI complexity.
- **Translation runs in interactive Claude sessions** instead of unattended. Slower for the user (must invoke Claude), but quality on prose is preserved by the implicit human review gate.

## Out of Scope (Explicit)

- Translating blog posts, publications, glossary, photography, changelog.
- Auto-detecting browser language and redirecting.
- Currency, date, or number format localization.
- Right-to-left support.
- A CMS or translation memory database.
- Per-paragraph stable IDs.
- A GitHub Action calling the Claude API.
